검증·거버넌스 솔루션

AI가 화면을 직접 생성하는 시대, UI 컴포넌트 판매 모델은 한계에 다다랐다. 살아남는 길은 생성과 경쟁하는 것이 아니라, 생성된 화면이 병원에 배포되기 전 반드시 통과해야 하는 "검증 레이어"를 소유하는 것이다.

목적 신규 사업 방향 보고 대상 시장 의료 AI · EHR 화면 검증 핵심 자산 검증 룰셋 IP
01 · WHY

필요성 — 왜 지금인가

시장의 무게중심이 "화면을 만드는 일"에서 "만들어진 화면을 믿을 수 있는가"로 이동하고 있다.

사라지는 시장

UI 컴포넌트 수요의 구조적 감소

그리드·차트·폼을 배치하는 "화면을 그리는 노동"을 이제 LLM이 대신한다. 컴포넌트를 더 잘 만드는 문제가 아니라, 상품 범주 자체의 수요가 줄어든다.

커지는 수요

검증 병목의 급증

AI가 화면을 빠르게 찍어낼수록 "이게 안전한가"를 보증하는 부담이 커진다. 특히 의료 EHR 화면은 오류가 곧 환자 안전 문제 — 생성 AI가 스스로 보증하지 못하는 영역이다.

→ 전략적 결론

생성과 정면 경쟁하지 않는다. 생성된 화면이 병원에 배포되기 전 반드시 통과하는 "검증 게이트"를 소유한다. 이 게이트가 고객의 배포 파이프라인에 내재화되는 순간, 단순 납품이 아니라 대체하기 어려운 구조가 된다.

02 · HOW

접근방법 — 어떻게 만드는가

핵심 원리는 엔진과 룰의 분리다. 검증 엔진은 도메인 지식 없이 지금 만들고, 의료 규제 룰은 현장에서 점진적으로 축적해 그 위에 얹는다.

01
입력 어댑터
AI 출력 형식(HTML·React·JSON)을 받는 입구. 실제 쓰이는 형식 하나부터 시작.
02
공통 중간표현(IR)
입력을 단일 구조 트리로 변환. 모든 규칙이 이 위에서 돌아 형식 변동에 안정적 — 기술 핵심.
03
규칙 엔진
검사 규칙을 플러그인으로. 심각도별 분류로 배포 차단 여부 결정.
04
리포트 + 배포 게이트
개발자용 위반 리포트와, CI에서 통과/차단을 판정하는 게이트.
05
룰 저장소 (IP)
현장에서 축적한 검증 룰이 쌓이는 곳. 회사의 독자 자산.
접근성객관 일관성객관 데이터 정합성반자동 규제·안전도메인
0–3개월
1단계 · 관통 프로토타입
어댑터 1종 + IR + 접근성 규칙 + 리포트. 도메인 지식 없이 즉시 착수 가능한 저비용 구간.
목표 — "화면을 넣으면 위반 리포트가 나온다" 데모로 신뢰 확보(입장권).
3–9개월
2단계 · 게이트화 + 검사 확장
리포트를 CI 배포 게이트로 전환, 일관성·데이터 검사 추가. 고객 파이프라인에 내재화.
목표 — 배포 프로세스에 박혀 "대체하기 어려운 구조" 형성.
9개월~
3단계 · 규제 룰 축적 (해자)
현장의 규제·안전 요건을 룰로 코드화해 축적. 따라 만들기 어려운 도메인 깊이 확보.
목표 — 룰 저장소를 회사만의 독자 자산으로 전환.
03 · 결론

결론 및 제언

방향은 명확하다. 작게 빠르게 시작하되, 진짜 자산은 룰셋 IP로 모은다.

  • 생성이 아니라 검증을 소유한다. 생성 영역은 빠르게 코모디티화되지만, "배포 전 반드시 통과하는 신뢰 게이트"의 수요는 AI가 빨라질수록 커진다.
  • 엔진은 지금, 룰은 현장에서. 도메인 지식 없이도 1단계는 즉시 착수 가능하며, 의료 규제 룰은 사업을 진행하며 축적해 해자로 만든다.
  • 진짜 경쟁력은 룰 저장소다. 엔진은 공개돼도 무방하나, 축적된 검증 룰셋은 회사가 소유해야 독자 자산이 된다.
양보 불가 원칙

검증 룰셋 IP는 소프트보울에 귀속한다

공동 개발을 하더라도 축적한 검증 룰셋의 권리는 소프트보울 귀속을 계약에 명문화한다. 이 조항이 없으면 시간을 들여 만든 해자를 협력사에 헌납하고, 다시 관계만 남는 출발점으로 돌아간다.

다음 단계 제언

1.1단계 데모 우선 착수. 저비용·고효과의 "입장권" — 의사결정과 무관하게 만들어 둘 가치가 있다.
2.룰 IP 귀속을 협상 테이블의 전제로. 파트너십 논의 초기에 권리 귀속을 명확히 한다.
3.3단계 본격 투자는 핵심 전제 확인 후 결정. 파트너의 검증 외주 의향·시장 환경을 점검한 뒤 단계적으로 확대한다.
