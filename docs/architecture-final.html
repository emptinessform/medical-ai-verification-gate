소프트보울 의료 AI 생성물 검증·거버넌스 솔루션 — 확정 최종 시스템 아키텍처

기반 결정: A(출하우선)을 골격으로, C(거버넌스해자)의 책임경계·층화 코어를 구조에 접목하고, B(진화확장)의 타입 규율을 흡수한 정당화된 하이브리드. 세 렌즈가 공통으로 권고한 절충선을 그대로 채택한다 — 스키마/인터페이스는 3단계 미래형으로 1일차에 동결하고, 실행코드/인프라는 1단계 현재형으로 미룬다. 이 한 문장이 제약 A(무재작업 진화)와 1단계 출하속도를 동시에 만족시키는 유일한 규율이다.

확정본 변경 요약(적대적 검토 반영): 본 버전은 v1의 구조·강점을 유지하되, 검토가 지적한 횡단 실행계약 공백을 모두 메웠다 — (1) fail-closed가 status(unknown/ambiguous)까지 지배하도록 유효심각도 모델을 재정의(§4.4, ADR-5), (2) ContractBinder 단계 신설로 계약기반 정합성을 실행가능하게 함(§3.6, ADR-3·9), (3) 결정적 nodeId 생성·교차시나리오/재실행 안정성 규약 확정(§3.2.1, ADR-10), (4) Promoter↔facts 순서 모순 해소(§1.2 파이프라인 재정렬), (5) scenarios dedup을 1단계로 끌어와 명세(§3.7), (6) exit-code 계약·prevHash·tenant 합성·미측정 게이트·break-glass 인가·위양성 라벨링을 1일차 고정(§5.2·§6.2·§4.7·§7). 모든 변경은 "필드/단계/규약" 추가이며 인터페이스 동결 원칙을 깨지 않는다.

#0. 요약 & 설계 원칙

#0.1 무엇을 짓는가 (한 문단)

AI(LLM)가 생성한 의료 EHR 화면을 병원 배포 전에 검사하는 게이트다. 생성과 경쟁하지 않고 "배포 전 반드시 통과하는 검증 레이어"를 소유한다. 5계층(어댑터 → IR → 규칙엔진 → 리포트+게이트 → 룰저장소)으로 구성되며, 소프트보울은 의학적 판정을 하지 않고 외부 승인 규격 위반 여부만 집행한다.

#0.2 기반 선정의 논리 (단순 짜깁기가 아닌 이유)

세 렌즈의 권고가 갈렸다 — 렌즈1(실현가능성)은 A 기반, 렌즈2(기술견고성)는 C 기반, 렌즈3(전략해자)은 C 기반. 이 충돌을 다음 원리로 해소한다.

  • 골격은 A로 잡는다. 세 렌즈가 일치한 사실: 무재작업 진화 구조(빈 L3 + requires 자동 skip + 바인딩 슬롯 + 순수함수 룰의 서버화 유보)는 세 안이 구조적으로 동등하게 정직하다. 그렇다면 1단계 출하속도가 가장 빠른 A가 골격의 자연스러운 기준점이다. C가 1단계로 당긴 머시너리(별도 Capture 단계, tenant_id/해시체인 감사 인프라)는 산문상 데모를 늦추는 과도설계다.
  • 그러나 C의 두 가지는 "필드/훅"만 추가하면 되는 저비용·고가치라 1일차에 접목한다. (a) 책임경계의 전역 fail-closed 불변식(authority 없으면 판정 출력 금지) — 나중에 못 끼워넣는 종류, SaMD 회피의 사업 존립 조건. (b) Capture/LIFT 층화(어댑터 물성이 L1로 누수되는 것을 한 겹 더 봉인) — 렌즈2가 집어낸 세 안 공통의 가장 심각한 누수 추상화를 메우는 유일한 해법.
  • C가 1단계로 당긴 인프라(해시체인 감사 저장, 멀티테넌트 격리 실물, 서버측 평가 노드)는 A식 규율로 2단계로 미룬다. 스키마 필드만 1일차에 박고 구현은 유보 → C의 전략 가치를 출하속도 손실 없이 확보.
  • B의 타입 규율은 비용 0이라 전면 흡수한다. unknown ≠ null 분리, irSchemaVersion 단일 패키지 공유, 룰셋 version pin.

#0.3 핵심 설계 원칙 (P1~P10)

# 원칙 구조적 강제 방식 (말이 아닌 코드)
P1 스키마는 미래형, 코드/인프라는 현재형 (제약 A) IR L1/L2/L3·바인딩 슬롯·authority·tenantId·rollout enum을 1일차 타입으로 동결. eval-server/DB/해시체인 저장/큐는 인터페이스 뒤로 유보
P2 어댑터는 Capture만, IR은 LIFT가 만든다 (제약 C·G) InputAdapter.capture()→Capture, 별도 lift(Capture)→L1. 어댑터 물성(런타임/정적/선언적)을 Capture에 가두고 LIFT가 균질 L1 생성
P3 규칙은 IR+facts만 읽는 순수 함수 (제약 F) Rule.evaluate(ir, ctx): Violation[] — I/O·네트워크·부수효과 금지. 로컬→서버 격리가 인터페이스 변경 0
P4 소프트보울은 판정하지 않고 집행한다 (제약 E) 전역 fail-closed: authority 없는 Finding은 스키마 검증에서 거부. 차단 정책은 tenantPolicy가 소유
P5 추론으로 배포를 막지 않는다 (제약 E) confidence-gating + status-gating: IR 노드 confidence < rule.confidenceFloor 또는 status∈{unknown,ambiguous}면 block→warn 강등(룰별 unknownPolicy가 지배). L1 관측사실(1.0/known)만 강하게 차단
P6 unknown은 null과 다르다 (제약 B) TriBool/status:'unknown'|'ambiguous' 1급이며 게이트까지 전파된다. 정적 파서가 못 푼 분기를 false로 단정하는 거짓 통과/위반을 타입·게이트 양단에서 차단
P7 결정성 = 감사성 (제약 E·H) MountSpec(seed/fixtures/waitFor) + 결정적 nodeId(§3.2.1) + inputDigest + 룰셋 version pin. 같은 (입력, 룰셋버전, IR스키마버전) → 같은 IR → 같은 결정
P8 멀티테넌시는 day-0 1급 키 (제약 G) tenantId가 IR 루트·Capture·룰셋·게이트결정·감사의 타입 강제 필드. 런타임 규약 아님
P9 심각도(코드) ≠ rollout(거버넌스) (제약 H) 유효심각도 = rule.defaultSeverity × tenant.rolloutState(shadow→warn→block) × confidence/status 강등. break-glass는 block-overridden 게이트 상태
P10 외부 계약은 규칙 밖에서 IR에 주입한다 (제약 D·F) ContractBinder 단계가 contractCatalog를 읽어 binding.contract를 채운다. 규칙은 순수성을 유지하고 계약 조회 I/O를 하지 않는다

#0.4 1단계 진짜 롱폴 리스크 격상 (세 렌즈 공통 교정)

세 안 모두 "빈 L3"를 헤드라인으로 삼았으나, 실제 1단계 롱폴 리스크는 L1→L2 Semantic Promoter다 (off-the-shelf 불가, day-1 필수, required/intent/label 휴리스틱이 confidence 값의 출처인데 세 안 모두 미명세). 본 문서는 이를 1순위 리스크로 격상하고 (§3.5) 알고리즘을 명세하며, confidence/unknown을 그 출력의 1급 필드로 강제하고, 입력형식별 어댑테이션 훅을 남겨 FHIR/DOM 시맨틱 분기로 인한 2단계 잠재 재작업을 선제 흡수한다.

#1.2 종단 데이터 흐름 (1 Verification Run)

데이터는 **단방향(왼→오)**으로만 흐른다. 어댑터는 규칙을 모르고, 규칙은 어댑터를 모른다 — 둘 다 IR만 안다.

검토 교정(Promoter↔facts 순서 모순 해소): v1은 4.PROMOTE 가 6.NORMALIZE 보다 앞서면서도 Promoter가 facts를 소비했다(promote(l1, facts, hook)). 확정본은 외부엔진 정규화(NORMALIZE)를 LIFT 직후, PROMOTE 직전으로 당긴다. Promoter는 항상 완성된 facts[]를 입력으로 받는다. 또한 계약기반 정합성을 위해 BIND(ContractBinder) 단계를 PROMOTE 뒤에 신설한다.

1. INGEST    artifact + RunContext{tenantId, repo, commit, ruleSetPin} 수신 → Run 생성
                inputDigest = sha256(정규화된 아티팩트 바이트 ‖ ruleSetPin ‖ irSchemaVersion)
2. CAPTURE   어댑터 detect() → capture(): Capture{domSnapshot|ast|declarative}
                런타임 경로면 Playwright 마운트 (MountSpec로 비결정성 봉인)
                scenarios[]: 빈 폼/검증에러/로딩 등 복수 상태 캡처 (제약 C 상태폭발 대응)
3. LIFT      Capture → L1 StructureGraph (어댑터 물성을 여기서 균질화, 결정적 nodeId 부여 §3.2.1)
4. NORMALIZE 외부엔진(axe-core 등) 실행 → facts[]:ExternalFact (L1 nodeId에 결합, measurable 플래그)
5. PROMOTE   (L1, facts) → L2 SemanticOverlay (Semantic Promoter: 폼·역할·바인딩 추론 + confidence/status)
6. BIND      ContractBinder(L2, contractCatalog[tenant]) → binding.contract 채움 (scope=contract 승격)
                계약 미발견·미매칭은 binding.contract.status='unknown' (fail-open 아님: 정합성 룰이 unknown 처리)
7. ANNOTATE  L2 → L3 DomainAnnotation (3단계만; 1단계 no-op, 빈 배열)
8. EVALUATE  활성 룰셋(tenant별, version-pinned) → 각 Rule.evaluate(IR, ctx) 순수 평가
                requires 미충족 레이어 의존 룰 자동 skip / confidence+status gating 적용
                모든 Violation에 authority 부착 (없으면 fail-closed)
9. DEDUP     교차 시나리오 Finding 합성·중복제거 (§3.7) → 1 Finding = (ruleId, stableNodeKey)
10. GATE     유효심각도 = severity × rolloutState × (confidence/status 강등), break-glass 적용 → Gate Decision
11. EMIT     Report(개발자용 + SARIF) + CI exit code(§5.2) + 불변 Audit 레코드(prevHash 체인)

핵심 불변식: 3~7에서 생성된 IR은 8에서 불변(read-only). 규칙은 IR을 변경 못 한다. 모든 Violation은 10으로 가기 전 authority(외부 권위 인용)를 갖는다 — 없으면 파이프라인 fail (fail-closed on missing provenance).

#2. L1 입력 어댑터

#2.1 책임 경계 (Capture/LIFT 층화 — C에서 접목)

어댑터의 유일한 책임은 정규화된 Capture 생성이다. 어댑터는 IR(L1)을 만들지 않는다. 이는 렌즈2가 집어낸 세 안 공통의 가장 심각한 누수 추상화를 메운다: A/B처럼 "어댑터=L1생성"으로 두면 L1Node.contrastRatio·a11y 같은 필드가 "런타임 경로에서만 채워지는" 조건부 필드가 되어 어댑터 물성이 L1 스키마의 채워짐/비워짐 패턴으로 직접 누수된다. Capture 중간 경계 한 겹이 이를 봉인하고, LIFT가 균질한 L1을 생성한다.

#2.2 인터페이스 (1일차 동결, 이후 불변)

interface InputAdapter {
  readonly id: string;                    // "dom-snapshot" | "fhir-questionnaire" | "tsx-static"
  readonly kind: CaptureKind;             // 'runtime-dom' | 'declarative' | 'source-ast'
  readonly priority: number;              // 동일 입력 다중 매칭 시 선택 순서

  detect(artifact: ArtifactDescriptor): MatchScore;   // 0..1, 0 = 처리 불가

  capture(artifact: ArtifactRef, ctx: CaptureContext): Promise<CaptureResult>;  // L1 아님, Capture만
}

// 검토 교정: capture는 실패를 throw가 아닌 값으로 반환 → exit-code 2(도구오류) 결정성 확보
type CaptureResult =
  | { ok: true; capture: Capture }
  | { ok: false; error: CaptureError };   // mount-failed | build-broken | unsupported | timeout

interface CaptureError { code: CaptureErrorCode; message: string; partial?: Capture; }
type CaptureErrorCode = 'mount-failed' | 'build-broken' | 'unsupported' | 'timeout' | 'detect-miss';

type ArtifactRef =
  | { type: 'url'; url: string; mount?: MountSpec }
  | { type: 'html'; html: string }
  | { type: 'bundle'; entry: string; files: VirtualFile[]; mount?: MountSpec }
  | { type: 'tsx-source'; files: VirtualFile[] }
  | { type: 'json-spec'; spec: unknown; schemaHint?: 'fhir-questionnaire' };

// 비결정성 봉인 (A에서 접목 — P7) — 재현성 = 감사 가능성
interface MountSpec {
  seed: number;                           // 난수 시드 고정 (기본 42)
  fixtures?: Record<string, unknown>;     // API/데이터 모킹
  viewport: { width: number; height: number };
  waitFor: 'networkidle' | { selector: string } | { ms: number };
  locale?: string;
}

// 복수 상태 캡처 (C에서 접목 — 상태폭발 대응, 제약 C)
interface CaptureContext {
  tenantId: TenantId;                     // P8: day-0 1급
  mount?: MountSpec;
  scenarios: ScenarioSpec[];              // [{id:'empty'}, {id:'validation-error', interactions:[...]}, {id:'loading'}]
  timeoutMs: number;
}

interface Capture {
  captureKind: CaptureKind;
  tenantId: TenantId;
  scenarios: CapturedScenario[];          // 상태별 캡처 — 미캡처 분기는 LIFT가 unknown 노드로 보완
  captureProvenance: {                    // 감사: 이 캡처가 어떻게 만들어졌나
    adapterId: string; adapterVersion: string;
    method: 'rendered' | 'parsed' | 'declarative';
    engine?: string;                      // "playwright@1.4x / chromium-12x"
    mountSpecDigest?: string;             // 재현성 키
  };
}

interface CapturedScenario {
  scenarioId: string;
  domSnapshot?: SerializedDOM;            // 런타임: outerHTML + computedStyles + a11yTree + focusOrder
  ast?: SourceAST;                        // 정적: best-effort AST
  declarative?: JsonDocument;             // 선언적: FHIR Questionnaire 등
  stablePath: StablePathMap;              // 각 캡처 노드 → 결정적 안정경로(§3.2.1) — nodeId 생성 입력
  sourceMap: SourceMap;                   // 각 캡처 노드 → 원본(file:line:col 또는 domPath)
}

#2.4 어댑터 추가 방법

  1. InputAdapter 구현 (id/kind/priority/detect/capture).
  2. 레지스트리 등록 (pnpm 워크스페이스 자동 디스커버리).
  3. 골든 픽스처(입력 → 기대 Capture, 결정적 nodeId 포함) 추가. → IR·LIFT·규칙 코드 변경 0 (제약 G). detect 점수 최고 어댑터 선택, 동률/저신뢰는 unknown provenance 남기고 명시적 CaptureResult{ok:false} 반환(예외 throw 금지 — exit-code 결정성).

#3. L2 IR — 다층 스키마 + 구체 예시 (설계의 심장)

#3.1 다층 구조 (제약 B: 단일 평면 트리 금지)

세 레이어는 별도 그래프이며 nodeId 참조로만 결합한다. 한 노드가 렌더구조·폼시맨틱·바인딩·의료시맨틱을 동시에 들지 않는다.

L1 StructureGraph   : 렌더 구조 사실(fact)만. LIFT가 생성. confidence=1.0(관측)
        │ nodeId 참조
        ▼
L2 SemanticOverlay  : 폼/역할/상호작용/바인딩. Promoter 추론 + ContractBinder 보강. confidence<1 가능
        │ nodeId 참조
        ▼
L3 DomainAnnotation : 의료 시맨틱(코드체계·단위·규격대조). 1단계 빈 배열. 3단계 채움

#3.2 공통 메타 (모든 레이어 노드 1급 필드 — 제약 B)

type NodeId = string;
type TriBool = true | false | 'unknown';      // B에서 접목 — P6

interface NodeMeta {

  derivedFrom: NodeId[];         // C에서 접목 — 레이어 간 근거 체인 (provenance 역추적)

}

#3.2.1 결정적 nodeId 생성 규약 (검토 high-3 해소 — 조인/dedup 키의 토대)

nodeId는 레이어 결합·교차 시나리오 dedup·골든 스냅샷의 유일한 부하 지지 키다. 생성·안정성 규칙이 없으면 결정성과 골든 스냅샷이 무너진다. 다음을 1일차 동결한다.

  • 생성 함수(결정적, 순수): nodeId = "{layer}:{sha1(tenantId ‖ stablePath)}.slice(0,12)"
    • layer ∈ {l1,l2,l3}. L2/L3 노드는 자신이 derivedFrom하는 L1 노드의 stablePath를 시드로 사용해 결합을 결정적으로 만든다.
    • stablePath(StablePathMap)는 시나리오 불변(scenario-invariant) 구조 경로다. 런타임 DOM은 tag 시퀀스 + 안정 속성(id/name/data-testid) + 형제 인덱스의 정규화 경로(예: form>select[name=order.drug]). 시나리오별로 달라지는 휘발 속성(value, aria-expanded, 포커스 상태)은 경로에서 제외한다.
  • 교차 시나리오 안정성(stableNodeKey): 같은 구조 노드는 모든 scenario에서 **동일 nodeId**를 갖는다(시드가 시나리오 불변이므로). 이것이 §3.7 dedup의 키다. 시나리오에서만 등장하는 노드(예: validation-error에서만 나타나는 에러 메시지)는 자신의 stablePath로 고유 id를 얻고 scenarioCoverage로 출처가 추적된다.
  • 재실행 안정성: (입력, ruleSetPin, irSchemaVersion)가 같으면 nodeId 집합이 비트 동일. MountSpec.seed 고정이 DOM 순서 안정성을 보장한다.
  • 충돌 처리: 동일 stablePath가 둘 이상이면(드물게 익명 형제) 정규화 경로에 결정적 형제 서수를 append(...>div[2])해 충돌 해소. 무작위 카운터 금지(재실행 비결정성 유발).
  • 불안정 입력 표기: 안정 앵커(id/name/testid)가 전혀 없어 경로가 형제 인덱스에만 의존하는 노드는 nodeMeta.status에 영향은 없으나 provenance에 pathStability:'index-only'를 남겨 골든 스냅샷 검토자가 취약 경로를 식별하게 한다.

#3.3 레이어별 노드 타입

// ── L1: 구조 그래프 (LIFT 생성, 사실만) ──
interface L1Node extends NodeMeta {
  tag: string;                              // 'input','button','div'
  attributes: Record<string, string>;
  text?: string;
  // 측정값 — 가용성을 1급으로 (렌즈2 누수 교정): 정적 경로는 measured:false
  computed?: {
    contrast?: Measured<number>;            // {value, measured} — '미측정'과 '통과' 구분
    focusable?: Measured<boolean>;
    tabOrder?: Measured<number>;
    fontSizePx?: Measured<number>;
  };
  a11y?: { role?: string; name?: Measured<string> };
  children: NodeId[];
}
// 측정 가용성 1급화 — 렌즈2 미해결난제(2) 해소
interface Measured<T> { value: T | null; measured: boolean; }   // measured:false = 이 경로로 측정 불가
interface L1Graph { rootId: NodeId; nodes: Record<NodeId, L1Node>; }

// ── L2: 시맨틱 오버레이 (폼·역할·바인딩) ──
type L2Overlay = NodeMeta & (
  | { kind: 'Form'; fields: NodeId[]; actions: NodeId[] }
  | { kind: 'Field'; label: string | null; required: TriBool;
      dataType: 'string'|'number'|'date'|'code'|'quantity'|'boolean'|'unknown';
      binding?: BindingDescriptor }
  | { kind: 'Action'; role: 'submit'|'cancel'|'destructive'|'navigate'|'unknown'; target?: string }
  | { kind: 'Display'; binding?: BindingDescriptor }
);

// ── 바인딩 디스크립터 — 단위·코드체계 슬롯 (제약 D) ──
interface BindingDescriptor {
  scope: 'ui-internal' | 'contract';        // 폐쇄형 vs 계약기반 공식 분리 (제약 D)
  path: string;                             // "order.dose"
  observedType?: string;                    // 폐쇄형 정합성용 (UI 내부 관측)
  unit?: { system: 'UCUM'; code?: string; status: 'known'|'unknown' };           // UCUM 슬롯
  codeSystem?: { system: 'LOINC'|'RxNorm'|'SNOMED'|'ICD10'; code?: string; status: 'known'|'unknown' };
  // ContractBinder가 채우는 슬롯 (§3.6) — Promoter는 비워둠
  contract?: {
    schema: 'FHIR'|'OpenAPI'; resource: string; element: string;
    status: 'known' | 'unknown';            // 계약 미발견/미매칭 = unknown (정합성 룰이 처리)
    contractRef?: string;                   // 카탈로그 내 계약 식별자 + 버전
  };
}

// ── L3: 도메인 주석 (3단계, 1단계 빈 배열) ──
type L3Annotation = NodeMeta & {
  appliesTo: NodeId;                        // L2 노드 참조
  clinicalConcept?: { system: string; code: string; display: string };
  contractMatch?: {
    contractRef: string;
    matched: boolean;
    authority: AuthorityRef;                // ★ 외부 권위 강제 (제약 E)
  };
};

#3.4 외부 사실(facts) 스키마 (axe→facts 매핑 명세 — 검토 thin 해소)

interface ExternalFact {
  engine: string;                           // "axe-core@4.x"
  ruleId: string;                           // 엔진 native 룰 id ("label","color-contrast")
  appliesTo: NodeId;                        // L1 nodeId — axe target → DOM node → stablePath → nodeId
  impact: 'minor'|'moderate'|'serious'|'critical';
  measurable: boolean;                      // 이 입력 경로에서 측정 가능했나 (정적=false 가능)
  observed: Record<string, unknown>;        // {accessibleName:""} | {ratio:1.6, required:4.5}
  scenarioId: string;                       // 어느 상태에서 관측됐나
}

axe→facts 매핑 규약(NORMALIZE 단계): axe-core 결과의 각 node.target(CSS 셀렉터)을 캡처 시점의 DOM 노드로 역참조하고, 그 노드의 stablePath로 L1 nodeId를 얻어 appliesTo에 결합한다. axe target이 복수 노드를 가리키면 각각 별도 fact로 분해한다. axe incomplete(판정 불가) 결과는 measurable:true, observed:{result:'incomplete'}로 보존하여 규칙이 unknown으로 처리하게 한다 — 누락(silent drop) 금지.

#3.5 1순위 리스크: Semantic Promoter 알고리즘 명세 (세 안 공통 사각 해소)

L1→L2 추론은 day-1 필수이며 off-the-shelf가 불가능하다. 입력은 항상 (L1, facts) — facts는 NORMALIZE에서 이미 완성됨(§1.2 순서 교정).

interface SemanticPromoter {
  promote(l1: L1Graph, facts: ExternalFact[], hook: PromoterHook): L2Overlay[];
}
// 입력형식별 어댑테이션 훅 — FHIR/DOM 시맨틱 분기로 인한 2단계 재작업 선제 흡수
interface PromoterHook {
  captureKind: CaptureKind;
  // FHIR Questionnaire는 required/dataType가 명시적(confidence 1.0), DOM은 휴리스틱(confidence<1)
  fieldRequiredHint?(node: L1Node): TriBool;
  dataTypeHint?(node: L1Node): string | undefined;
}

추론 규칙 (DOM 경로, 각각 confidence·status 산출):

  • Field 식별: tag∈{input,select,textarea} 또는 role∈{textbox,combobox,spinbutton,checkbox} → Field. confidence 0.95, status known.
  • required: attr.required 또는 aria-required=true → true(0.95, known); 인접 텍스트에 */"필수" → true(0.7, known); 신호 없음 → 'unknown'(미상, false 단정 금지 — P6), status ambiguous.
  • label: <label for> 연결(0.95) > aria-label(0.9) > aria-labelledby(0.9) > 인접 텍스트 휴리스틱(0.6, status ambiguous) > 없음 → null + status known(확정 부재, axe label fact와 교차검증).
  • dataType: input[type] 직매핑(0.9); inputmode/pattern 보조(0.7); 불명 → 'unknown', status ambiguous.
  • Action.role: type=submit 또는 폼 내 단독 버튼(0.85); 텍스트 "삭제/취소/저장" 사전(0.7); 불명 → 'unknown', status ambiguous.
  • binding.path: name/data-bind/id 속성에서 추출(0.8); 없음 → binding 생략.

핵심 규율: 추론 불가는 반드시 'unknown'/status:'ambiguous'로 표기하고 confidence를 낮춘다. 이 두 신호는 §4.4의 effectiveSeverity로 직결되어 "추론으로 배포를 막지 않음"을 보장한다(P5).

#3.6 ContractBinder — 계약기반 정합성의 실행 경로 (검토 high-2 해소)

검토가 지적한 결함: ADR-3은 "Promoter가 계약 스키마를 IR에 주입한다"고 했으나 promote()는 계약을 입력으로 받지 않고 어느 단계도 binding.contract를 채우지 않아 계약기반 정합성이 구현 불가였다. 확정본은 계약 주입 책임을 Promoter에서 분리해 전용 ContractBinder 단계로 이관한다(파이프라인 6.BIND). Promoter는 UI 관측만, Binder는 외부 계약 결합만 — 단일 책임이 명확해진다.

// contractCatalog: tenant별 외부 계약 묶음 (FHIR 프로파일/OpenAPI). 로드 소스는 룰저장소(L5)와 동일 경로.
interface ContractCatalog {
  tenantId: TenantId;
  version: string;                          // 카탈로그 version pin (감사·재현성)
  contracts: ContractEntry[];
}
interface ContractEntry {
  ref: string;                              // "fhir-r4:MedicationRequest@hospA-1.2"
  schema: 'FHIR' | 'OpenAPI';
  resource: string;                         // "MedicationRequest"
  elements: ContractElement[];              // path → {type, unit, codeSystem, required}
  authority: AuthorityRef;                  // ★ 계약 자체의 외부 권위 (병원/제조사 승인)
}

interface ContractBinder {
  // 순수 함수: L2 + 카탈로그를 읽어 contract 슬롯을 채운다. 네트워크 I/O 없음(카탈로그는 미리 로드됨).
  bind(l2: L2Overlay[], catalog: ContractCatalog): L2Overlay[];
}

scope→contract 승격 규칙(결정적):

  1. Binder는 각 Field/Display의 binding.path를 카탈로그 elements[].path와 정규화 매칭(케이스/구분자 무시, 별칭 테이블 적용)한다.
  2. 매칭 성공: binding.scope = 'contract'로 승격, binding.contract = {schema, resource, element, status:'known', contractRef}, 계약이 명시한 unit/codeSystem/required를 슬롯에 채우되 Promoter 관측값과 충돌하면 둘 다 보존(정합성 룰이 불일치를 위반으로 발화).
  3. 매칭 실패(계약에 없는 path) 또는 카탈로그 부재: binding.scope는 ui-internal 유지, binding.contract.status = 'unknown'. fail-open 아님 — 계약기반 정합성 룰은 status:'unknown'을 "계약 미확인"으로 별도 발화(차단 아닌 경고, P5와 일관).
  4. 카탈로그 version은 inputDigest 산입 대상에 포함되어 카탈로그 변경이 재현성에 반영된다.

로드 소스: contractCatalog는 룰저장소(L5)와 같은 메커니즘으로 공급된다 — 1단계 Git manifest, 2단계 Postgres + 서명 번들. 1단계에는 카탈로그가 비어 있고(contracts:[]) BIND는 전 노드를 contract.status:'unknown'으로 통과시킨다(계약기반 룰은 2단계에 활성). 슬롯·단계가 1일차에 존재하므로 무재작업.

#3.7 교차 시나리오 Finding 합성·dedup (검토 R2 — 1단계로 끌어옴)

검토 교정: v1은 scenarios dedup을 2단계로 미뤘으나, scenarios[]가 1단계 기능이므로 미명세 시 1단계 리포트가 즉시 중복으로 오염된다. 확정본은 dedup 규약을 1일차 명세로 끌어온다.

  • dedup 키: (ruleId, stableNodeKey) — stableNodeKey는 §3.2.1의 시나리오 불변 nodeId. 같은 룰이 같은 구조 노드에서 여러 scenario에 발화하면 1 Finding으로 병합.
  • evidence 합성: 병합된 Finding의 evidence.observed는 시나리오별 관측을 배열로 보존(perScenario: [{scenarioId, observed}]). 리포트는 "3개 상태 중 2개에서 발화" 같은 커버리지를 표시.
  • 유효심각도 합성: 병합 Finding의 effectiveSeverity는 시나리오별 effectiveSeverity의 최댓값(가장 강한 차단) 으로 정한다. 단 어떤 시나리오에서 노드 status가 known이고 다른 시나리오에서 unknown이면 known 관측을 우선한다(관측 우위).
  • 시나리오 고유 위반: 특정 상태에서만 존재하는 노드의 위반(예: validation-error에서만 나타나는 잘못된 에러 카피)은 고유 nodeId이므로 병합되지 않고 단독 Finding으로 남는다 — scenarioCoverage가 출처를 명시.
  • 결정성: dedup은 정렬된 (ruleId, nodeId) 순회로 수행 → 같은 입력에 같은 병합 결과(골든 스냅샷 안정).

#4. L3 규칙 엔진 — 플러그인 인터페이스 + 심각도/롤아웃

#4.1 규칙 플러그인 인터페이스 (시그니처)

interface Rule {
  readonly id: string;                    // "a11y.input-label"
  readonly category: 'accessibility' | 'consistency' | 'data-integrity' | 'regulatory-safety';
  readonly version: string;               // semver — 감사·롤아웃 단위
  readonly defaultSeverity: Severity;     // 'block' | 'warn' | 'info'
  readonly requires: IRLayer[];           // ['l1'] | ['l2','facts'] | ['l2','l3'] → 미충족 시 자동 skip
  readonly confidenceFloor: number;       // 이 미만 confidence면 block→warn 강등 (P5)
  readonly unknownPolicy: UnknownPolicy;  // ★ status가 unknown/ambiguous일 때 동작 (검토 high-1)
  readonly authority: AuthorityRef;        // ★ 필수 (optional 아님) — fail-closed (제약 E)

  // 순수 함수: IR+facts 읽기만, I/O·네트워크·부수효과 금지 (P3)
  evaluate(ir: Readonly<IR>, ctx: RuleContext): Violation[];
}

// 검토 high-1 해소: 룰이 unknown/ambiguous 노드를 어떻게 다룰지 명시적 선언
type UnknownPolicy =
  | 'skip'        // 미상 노드는 평가 제외 (예: required가 unknown인데 required 검사면 발화 안 함)
  | 'demote'      // 발화하되 block→warn 강등 (불확실하나 알릴 가치)
  | 'report';     // status:unknown Finding으로 정보 보고 (info, 게이트 미반영)

type Severity = 'block' | 'warn' | 'info';
type IRLayer = 'l1' | 'l2' | 'l3' | 'facts';

// 외부 권위 — 모든 Finding 동반 필수 (제약 E)
interface AuthorityRef {
  ruleSource: string;        // "WCAG 2.2 AA" | "hospital-A-medication-spec" | "FHIR-R4"
  clause?: string;           // "1.4.3" | "MedicationRequest.dosage"
  version: string;
  approvedBy?: string;       // 도메인 룰(L3)은 병원/제조사 승인자 강제
}

interface Violation {
  ruleId: string; ruleVersion: string;
  severity: Severity;                      // 엔진이 rollout/confidence/status로 effective 재계산
  nodeId: NodeId;
  nodeStatus: NodeStatus;                  // ★ 평가 대상 노드의 status (게이트로 전파 — P6)
  message: string;
  provenance: Provenance;                  // 수정위치 = IR 노드 provenance 전파
  authority: AuthorityRef;                 // ★ 없으면 스키마 검증 거부 (fail-closed)
  evidence: {                              // A에서 접목 — 관측값 1급
    kind: 'fact' | 'computed' | 'structural';
    engine?: string;
    observed: unknown;                     // 무엇을 보고 판정했나 (예: {ratio:1.6, required:4.5})
  };
  confidence: number;                      // IR 노드 confidence 전파
  fixHint?: string;
}

#4.2 규칙이 IR 위에서 도는 방식 (예시)

// a11y.input-label — 1단계 5규칙 중 하나

규칙은 IR+facts만 읽는 순수 함수. requires에 l3을 선언한 규칙은 1단계에 L3가 없으므로 자동 skip(에러 아님) → 같은 엔진이 1·3단계 공존. 이것이 무재작업 진화의 메커니즘이자 룰 IP 서버화의 토대(P3).

#4.3 누수 교정: '미측정' vs '통과' 구분 (렌즈2 미해결난제 해소)

규칙은 Measured<T>.measured와 ExternalFact.measurable로 "측정 불가(정적 경로)"와 "측정됐고 통과"를 구분한다.

// a11y.color-contrast — 미측정을 위반으로 오판하지 않음
evaluate(ir) {
  return ir.l1nodes()
    .filter(n => n.computed?.contrast?.measured === true)         // 측정된 것만 평가
    .filter(n => n.computed!.contrast!.value! < 4.5)
    .map(n => violation(...));
  // contrast.measured===false (정적 경로) → 평가 대상 제외, status:unknown으로 리포트에 별도 표기
}

#4.4 심각도 모델 — 두 겹 × confidence × status 게이팅 (검토 high-1 핵심)

검토가 지적한 결함: v1의 effectiveSeverity는 숫자 confidence만 봤다. 그래서 status:'ambiguous'이면서 confidence가 floor 이상인 노드는 차단 동작이 미정의였고, P6(unknown 1급)이 게이트까지 도달하지 못했다. 확정본은 status를 effectiveSeverity의 1급 입력으로 추가하고, 룰별 unknownPolicy로 동작을 명시 선언하게 한다.

유효심각도(rule, binding, node) =
  rollout강등(  status강등( confidence강등( rule.defaultSeverity ) )  )

  ① confidence강등:  node.confidence < rule.confidenceFloor  ⇒  block → warn
  ② status강등:      node.status ∈ {unknown, ambiguous}      ⇒  rule.unknownPolicy 적용
                       - 'skip'   : Finding 제거 (게이트·리포트 미반영)
                       - 'demote' : block → warn (불확실하나 경고)
                       - 'report' : severity → info, rolloutState 무관 게이트 미반영
  ③ rollout강등:     binding.rolloutState=shadow ⇒ info ; =warn 이면 block → warn

interface RuleBinding {                   // 룰셋 안 룰 1개 배치 상태 (tenant별)
  ruleId: string; ruleVersion: string;
  rolloutState: 'shadow' | 'warn' | 'block';
  promotedBy?: string; promotedAt?: string;   // 승급 감사
}

function effectiveSeverity(v: Violation, binding: RuleBinding, rule: Rule): Severity | null {
  let sev = v.severity;

  // ① confidence-gating (P5)
  if (sev === 'block' && v.confidence < rule.confidenceFloor) sev = 'warn';

  // ② status-gating (P6 — 검토 high-1) — confidence가 floor 이상이어도 status가 미상이면 지배
  if (v.nodeStatus === 'unknown' || v.nodeStatus === 'ambiguous') {
    if (rule.unknownPolicy === 'skip')   return null;          // Finding 제거
    if (rule.unknownPolicy === 'report') return 'info';        // 정보만, 게이트 미반영
    if (rule.unknownPolicy === 'demote' && sev === 'block') sev = 'warn';
  }

  // ③ rollout-gating (P9)
  if (binding.rolloutState === 'shadow') return 'info';
  if (binding.rolloutState === 'warn' && sev === 'block') sev = 'warn';
  return sev;
}

불변식(게이트 차단 가능 조건): 어떤 Finding이 게이트를 차단하려면 반드시 nodeStatus==='known' 이고 confidence ≥ confidenceFloor 이고 rolloutState==='block'. 즉 미상·저신뢰·미승급 중 하나라도 있으면 절대 block에 도달하지 못한다. 이것이 "추론으로 배포를 막지 않는다"(P5)와 "unknown≠null이 게이트까지 산다"(P6)의 단일 강제 지점이다.

#4.5 Staged Rollout + Break-glass (제약 H)

  • Staged rollout: 신규 룰은 항상 shadow(데이터만 수집) → warn → block. 코드 변경 없이 룰셋 상태 전환만으로. 위양성률을 보고 승급.
  • confidence/status-gating: L1 관측사실(1.0/known)만 강하게 차단. L2/L3 추론·미상은 floor 미만 또는 unknown이면 경고/제거로 강등 → 오탐 신뢰붕괴와 SaMD 오인 동시 방지 (제약 E).
  • Break-glass: block 위반이라도 권한자가 사유+서명으로 우회 → 게이트는 block-overridden(통과)이지만 감사에 우회자·사유·타임스탬프가 1급. 최종 차단권은 병원 거버넌스(tenantPolicy가 소유).

#4.5.1 Break-glass 인가 모델 (검토 thin 해소)

break-glass는 인가된 주체만 수행 가능해야 무력화되지 않는다.

interface BreakGlassGrant {
  tenantId: TenantId;
  subject: string;                          // 우회 주체 (계정/서비스)
  role: 'governance-officer' | 'release-manager';   // tenantPolicy가 인가한 역할만
  scope: { ruleIds?: string[]; categories?: Rule['category'][] };  // 우회 가능 범위 제한
  expiresAt: string;
  signature: string;                        // tenant 거버넌스 키 서명 — 위조 방지
}
  • 게이트는 break-glass 요청 시 BreakGlassGrant 서명을 tenant 거버넌스 공개키로 검증한다. 검증 실패 시 우회 거부(차단 유지).
  • 규제/안전(regulatory-safety) 카테고리 룰은 tenantPolicy가 명시 허용하지 않으면 break-glass 불가(기본 금지) — SaMD 사정권에서 안전 위반을 무단 우회하는 경로 차단.
  • 모든 우회 시도(성공·실패)는 감사 레코드에 1급 기록.

#4.7 미측정 노드의 게이트 처리 (검토 thin 해소)

Measured<T>.measured===false(정적 경로 측정 불가) 또는 ExternalFact.measurable===false인 노드에 대해:

  • 측정 의존 룰(대비·포커스 등)은 해당 노드를 평가 대상에서 제외하고, 별도 coverage-gap 정보 Finding(info)을 생성해 "이 노드는 현재 입력 경로로 검사 불가"를 명시한다. 미측정을 통과로도, 위반으로도 처리하지 않는다.
  • 게이트는 coverage-gap을 차단에 산입하지 않으나, 리포트 summary.coverageGaps로 노출하여 정적 경로의 부분맹을 운영자가 인지하게 한다(런타임 경로 권장 신호).

  blocking: string[];                     // 차단 룰 ID들
  policy: { tenantPolicyVersion: string; blockOn: string };   // 차단 정책은 tenant 소유 (제약 E·H)

위반·근거·수정위치 3종 세트가 모든 Finding에 보장: 근거=evidence(관측값)+authority(외부 출처·버전·승인자), 수정위치=location(provenance 역추적한 원본 file:line:col 또는 domPath).

위양성 라벨링(검토 thin 해소): 운영자가 Finding을 confirmed-fp로 라벨하면 감사에 기록되고, 룰별 위양성률 메트릭(§7.3)에 집계되어 staged rollout 승급/강등의 입력이 된다. 라벨은 게이트 결과를 바꾸지 않으며(우회는 break-glass 전용), 룰 품질 피드백 루프에만 쓰인다.

#5.2 게이트 결정 로직 + Exit Code 계약 (검토 thin 해소 — 1일차 고정)

function decideGate(findings: ReportedFinding[], policy: TenantGatePolicy): GateDecision {
  const blocking = findings.filter(f =>
    f.effectiveSeverity === 'block' && f.rolloutState === 'block' && !f.overridden);

  if (blocking.length === 0) {
    const wasOverridden = findings.some(f =>
      f.effectiveSeverity === 'block' && f.rolloutState === 'block' && f.overridden);
    return { result: wasOverridden ? 'block-overridden' : 'pass', blocking: [], policy };
  }
  return { result: 'block', blocking: blocking.map(f => f.ruleId), policy };
}

Exit Code 계약 (1일차 동결, 단계 무관 고정):

code 의미 발생 조건
0 pass / 통과 gate.result ∈ {pass, block-overridden} (정상 통과 또는 인가된 우회)
1 block / 차단 gate.result==='block' (실제 위반으로 게이트 차단)
2 tool-error / 도구오류 마운트 실패·빌드 깨짐·미지원 입력·타임아웃(CaptureResult.ok===false)·스키마 검증 실패(authority 부재)
  • freeze 의미: 코드 0/1/2의 의미는 1단계부터 동결한다. 1단계는 리포트가 목표라 게이트를 비활성(--no-gate)으로 돌려도 exit code 매핑 자체는 동일하게 유지 — 2단계 게이트 전환 시 CI 워크플로 재작성이 불필요. code 2(도구오류)와 code 1(위반)을 구분하지 않으면 CI가 "도구 깨짐"을 "코드 위반"으로 오인하므로, 이 구분은 day-1 필수다.
  • code 2는 항상 비차단 실패로 취급할지(soft)·차단으로 취급할지(hard) 를 tenantPolicy.toolErrorMode가 결정한다(기본 soft: 도구 오류로 배포를 막지 않되 리포트에 경고).

#5.3 CI 통합

  • 1단계: CLI (softbowl verify --input <url|bundle> --tenant hosp-A --ruleset @pinned --out report.json). 리포트 + HTML 출력. 게이트는 --no-gate 기본이나 exit code 매핑은 §5.2대로 유지.
  • 2단계: 동일 CLI가 게이트 활성 → GitHub Actions/GitLab CI/Jenkins 한 스텝. SARIF 출력 추가 → PR 인라인 표시(위반이 소스 라인에 직접). CLI는 얇은 클라이언트, 룰 평가는 서버 API 호출(제약 F). 포맷터만 추가 → 재작업 없음.

#6.2.1 prevHash 체인 — 1단계 동작 정의 (검토 thin 해소)

v1은 "스키마는 1일차, 구현은 2단계"라고만 했다. 확정본은 1단계에서 prevHash가 실제로 무엇을 담는지 고정한다.

interface AuditRecord {
  auditRef: string;                         // 이 레코드 id
  tenantId: TenantId; runId: string;
  recordHash: string;                       // sha256(canonicalJSON(this 제외 prevHash 포함))
  prevHash: string;                         // 직전 (tenant별) 레코드의 recordHash, 최초는 "GENESIS"
  inputDigest: string; irHash: string;
  ruleSetVersion: string; contractCatalogVersion?: string;
  findings: ReportedFinding[]; gate: GateDecision;
  breakGlass?: BreakGlassGrant['signature'];
  createdAt: string;
}
  • 1단계 동작: 감사 레코드는 tenant별 append-only 파일에 기록되며 prevHash는 해당 tenant 직전 레코드의 recordHash를 실제로 담는다(체인이 1단계부터 형성됨, 빈 필드 아님). recordHash 계산은 1일차부터 수행한다. 차이는 검증 강제와 저장 매체뿐 — 1단계는 로컬/Git 파일에 신뢰 기반 저장, 사후 검증 스크립트 제공.
  • 2단계 활성: WORM object storage 이전 + 체인 무결성 자동 검증(파이프라인이 prevHash 일치를 확인, 불일치 시 변조 경보) + 서명 키 관리(HSM 여부는 R8에서 확정).
  • 체인 시드("GENESIS")와 정규화(canonicalJSON: 키 정렬·공백 제거) 규약을 1일차 고정 → 1·2단계 해시 호환.

#6.4 tenant 룰셋 합성 (검토 medium 해소 — 1단계 정의)

  • 활성 룰셋 = softbowl-global 베이스 + tenant-override 오버레이의 결정적 합성.
  • 합성 규칙: 같은 ruleId에 대해 tenant-override의 RuleBinding(rolloutState·promotedBy)이 global을 덮어쓴다. tenant에만 있는 룰은 추가, global에만 있는 룰은 유지. 충돌 해소는 항상 tenant 우선.
  • 1단계: 단일 tenant라 override가 비어 있어 합성 = global. 그러나 composedFrom 필드와 합성 함수는 1일차에 존재 → 멀티테넌트 전환 시 무재작업.
  • 3단계 멀티테넌트 거버넌스(override 승인 워크플로·우선순위 충돌 정책)는 R9에서 확정.

#7. 횡단 관심사

#7.1 책임 경계 / SaMD 회피 (제약 E) — 구조로 강제

  • 전역 fail-closed 불변식: 모든 Violation은 authority(외부 권위 인용) 없이 스키마 검증에서 거부된다. authority는 optional이 아닌 필수 필드. 시스템은 "우리가 판단했다"가 아니라 "규격 X 조항 Y와 불일치"만 출력한다.
  • status+confidence-gating: 불확실한 추론(L2/L3)·미상(unknown/ambiguous)으로 배포를 막지 않음 → "확신 없이 판정하는 자=SaMD 오인" 회피 (P5·P6, §4.4 단일 강제 지점).
  • 차단권 위임: 게이트 차단 정책은 tenantPolicy(병원)가 소유. 소프트보울은 정책을 집행만. 규제/안전 룰 break-glass는 tenant 명시 허용 없이 불가(§4.5.1). 포지셔닝 = "거버넌스·규약 준수 집행자"(CDSS/SaMD 아님).
  • 감사 1급: Finding.evidence+authority, break-glass 우회(인가 서명 포함), prevHash 체인이 스키마상 필수.

#7.2 멀티테넌시 · 벤더 독립 (제약 G)

  • tenantId가 IR 루트 필드로 타입 강제(P8) — RunContext뿐 아니라 IR·Capture·룰셋·게이트결정·감사·nodeId 시드의 1급 키. 런타임 규약이 아니라 컴파일타임 격리.
  • 룰셋은 softbowl-global + tenant-override 합성(§6.4). tenant별 rollout·게이트 정책·계약카탈로그·PHI 레닥션 강도 독립.
  • 어댑터는 표준 타깃(FHIR Questionnaire, 표준 React DOM)으로 일반화 — 특정 벤더(카카오) 전용 가정 없음. 단일 파트너 이탈해도 작동.
  • 격리 구현은 2단계, 키는 1일차: 1단계는 단일 tenant 사용이나 tenantId가 골격에 박혀 나중 리팩토링 지옥 회피.

#7.3 관측성

  • 구조화 로그(runId/traceId 상관). 메트릭: 룰별 발화율·위양성률(falsePositiveLabel 집계, 승급 입력), 어댑터 detect 신뢰도 분포, unknown/ambiguous 노드 비율·coverageGaps 수(IR 품질), 게이트 차단율·break-glass 빈도(거버넌스 건강성), scenario별 발화 차이.
  • 감사 ≠ 관측성 분리: 감사=법적 불변 기록(prevHash 체인), 관측성=운영 텔레메트리.
  • 결정성 회귀 테스트: 골든 IR 스냅샷(nodeId·dedup 결과 포함) — 같은 입력→같은 IR→같은 Finding 집합 보증.

#7.4 성능 예산 & 에러 처리 (검토 thin 해소)

  • 성능 예산: 1단계 Run은 단일 화면·복수 scenario를 동기 수초~수십초로 처리(Playwright 콜드 마운트 ~2–5s/scenario + axe ~수백ms + 룰 평가 <100ms). 목표 p95 < 30s/Run(3 scenario). scenarios·페이지 수가 늘면 2단계에 워커 풀로 병렬화(인터페이스 불변, Run은 멱등). 마운트는 scenario 간 브라우저 컨텍스트 재사용으로 콜드스타트 분할.
  • 에러 처리 분류: (a) 입력 결함(빌드 깨짐·미지원) → CaptureResult{ok:false} → exit 2. (b) 부분 캡처(일부 scenario만 성공) → 성공 scenario로 평가 진행 + coverage-gap 표기, exit은 게이트 결과 따름. (c) 룰 평가 내부 예외 → 해당 룰만 격리 실패(info 진단 Finding), 다른 룰·게이트는 계속(순수 함수라 격리 가능). (d) 스키마 검증 실패(authority 부재 등) → fail-closed, exit 2. 어떤 경우에도 침묵 통과(false pass) 금지.

#10. 핵심 설계 결정 (ADR) & 트레이드오프

ADR-1 — A를 골격 기반으로, C의 책임경계·층화를 1일차 접목 (하이브리드 정당화)

  • 결정: 무재작업 구조가 세 안 동등하므로 1단계 출하속도 최강인 A를 골격으로. C가 1단계로 당긴 인프라(별도 Capture 구현, tenant 격리 실물, 해시체인 저장)는 A식 규율로 2단계 유보하되, 필드/훅만 추가하면 되는 저비용·고가치 두 가지(전역 fail-closed authority, Capture/LIFT 층화)는 1일차 접목.
  • 대안: C 전면 기반(렌즈2·3) / A 순수(렌즈1). 근거: C 전면은 1단계 데모 속도 손실, A 순수는 SaMD 방어 코드강제 부족 + 어댑터 누수. 하이브리드가 교집합.
  • 트레이드오프: 하이브리드 통합 비용. 그러나 접목분이 전부 "스키마 필드/인터페이스 훅"이라 구현 부담 최소.

ADR-2 — 어댑터=Capture만, L1은 LIFT가 생성 (C에서 접목)

  • 결정: capture()→CaptureResult, 별도 lift()→L1. 대안: A/B의 buildL1=어댑터 책임.
  • 근거: 어댑터=L1생성이면 contrastRatio/a11y가 조건부 필드가 되어 어댑터 물성이 L1 스키마로 누수, 규칙이 '미측정'과 '통과'를 구분 못 함. Capture 한 겹이 봉인, Measured<T>.measured로 구분 1급화.
  • 트레이드오프: 변환 단계 +1. 정보 부재는 measured 플래그로 규칙 레벨 구분.

ADR-3 — 규칙은 순수 함수, 외부 계약은 ContractBinder가 IR에 주입 (A + 검토 high-2 교정)

  • 결정: 규칙은 IR·facts만 읽고 I/O 금지(requires 미충족 자동 skip). 계약 스키마 주입은 Promoter가 아니라 전용 ContractBinder 단계가 contractCatalog를 읽어 binding.contract를 채운다(§3.6).
  • 대안(기각): v1처럼 Promoter가 계약 주입 — Promoter는 UI 관측 책임이라 외부 계약 결합과 혼재하면 단일 책임 붕괴 + promote() 시그니처에 카탈로그가 없어 구현 불가였다(검토 지적).
  • 근거: 재현성(감사) + 테스트성 + 2단계 서버측 IP 격리 인터페이스 변경 0(제약 F). 계약 조회 I/O는 BIND 단계(미리 로드된 카탈로그)로 격리해 규칙 순수성 유지.
  • 트레이드오프: 파이프라인 단계 +1(BIND). 책임 분리 대가로 수용.

ADR-4 — 전역 fail-closed: authority 없는 판정 출력 금지 (C에서, 양보 불가)

  • 결정: Violation.authority 필수, 스키마 검증에서 부재 시 거부. 차단 정책은 tenantPolicy 소유.
  • 근거: 제약 E — SaMD/CDSS 사정권 회피의 사업 존립 조건. 런타임 규약/문서 약속으로는 부족, 컴파일·검증 단 fail-closed로 강제. 나중에 못 끼워넣는 결정.
  • 트레이드오프: 1단계 접근성 룰도 authority(WCAG 인용) 박는 보일러플레이트. 비용 무시 가능, 가치 결정적.

ADR-5 — confidence + status 게이팅: 추론·미상으로 차단 금지 (C에서, 검토 high-1로 강화)

  • 결정: effectiveSeverity가 confidence뿐 아니라 **status(unknown/ambiguous)**까지 1급 입력으로 받고, 룰별 unknownPolicy(skip/demote/report)로 동작을 선언한다. 게이트 차단 조건 = status known ∧ confidence≥floor ∧ rollout=block (§4.4 불변식).
  • 대안(기각): v1처럼 confidence만 — status:ambiguous ∧ confidence≥floor 노드의 차단이 미정의였고 P6이 게이트에 도달 못 함(검토 high-1).
  • 근거: 제약 E·B — 추론/미상으로 배포를 막으면 (a) 오탐 신뢰붕괴, (b) SaMD 오인, (c) unknown≠null 원칙이 게이트에서 무력화. 이를 단일 함수로 강제.
  • 트레이드오프: 일부 진짜 위반을 경고/제거로 강등(놓칠 위험). staged rollout·위양성 라벨링과 직교 결합해 보완.

ADR-6 — unknown ≠ null 타입 분리 + 단일 ir-schema 패키지 (B에서)

  • 결정: TriBool/status 1급. irSchemaVersion 박고 전 모듈이 @sb/ir-schema 공유. 대안: null로 미상 표현 / 모듈별 자체 타입.
  • 근거: 정적 파서가 못 푼 props 분기를 false로 단정하면 거짓 통과/위반. 타입 레벨 차단 + 게이트 전파(ADR-5)로 이중 강제. 단일 패키지가 경계 드리프트 컴파일타임 차단.
  • 트레이드오프: 모델 verbosity·전 패키지 재컴파일. 핵심 정합성 자산이라 수용.

ADR-7 — IR L1/L2/L3 다층, L3 1단계 빈 배열로 출하

  • 결정: 세 레이어 별도 그래프, nodeId 결합. L3는 1단계 []. provenance/confidence/status 공통 1급.
  • 근거: 빈 레이어 비용 0, 평면 트리 쪼개기 비용은 전 규칙 재작성. requires:['l3'] 자동 skip이 1·3단계 공존 보장.
  • 트레이드오프: 1단계 개발자가 안 쓰는 레이어 학습. 문서로 상쇄.

ADR-8 — 런타임 DOM 스냅샷 1차 + MountSpec 비결정성 봉인 + scenarios[] (A+C)

  • 결정: Playwright 마운트 1차, 정적 보조. MountSpec(seed/fixtures/waitFor) + scenarios.
  • 근거: 제약 C — 동적성·대비비·포커스·axe 재사용 동시 해결. MountSpec가 비결정성(1순위 운영 실패모드) 봉인 → 재현성=감사성. scenarios[]가 상태폭발 사각 대응.
  • 트레이드오프: 실행 가능 아티팩트(URL/번들) 전제, 마운트 비용. scenario 간 Finding 합성은 §3.7로 명세 완료(v1 미해결 항목 해소).

ADR-9 — 결정적 nodeId를 조인/dedup/스냅샷의 단일 키로 동결 (검토 high-3)

  • 결정: nodeId = layer:sha1(tenantId‖stablePath) — 시나리오 불변 stablePath 시드. 교차 시나리오·재실행 안정성을 §3.2.1로 규약화.
  • 대안(기각): v1의 무규약 nodeId — 생성·안정성 미정의로 결정성·골든 스냅샷·dedup이 모두 무너졌다(검토 high-3).
  • 근거: nodeId는 L1↔L2↔L3 결합, §3.7 dedup, §7.3 골든 스냅샷의 유일한 부하 지지 키. 비결정 시 전 하위 시스템 붕괴.
  • 트레이드오프: 안정 앵커 없는 노드는 인덱스 의존 경로(취약) — pathStability:'index-only'로 표기해 검토자가 식별.

ADR-10 — exit-code 계약·prevHash 체인·tenant 합성을 1일차 동결 (검토 medium 일괄 해소)

  • 결정: exit 0/1/2 의미(§5.2), prevHash 체인 동작(§6.2.1), 룰셋 합성 함수(§6.4)를 1단계부터 동작하는 형태로 고정(빈 필드·미래 약속 아님).
  • 근거: 이 셋은 "스키마만 박고 미루면" 2단계에 CI 워크플로 재작성(exit-code)·감사 호환성 붕괴(prevHash)·테넌트 리팩토링(합성)을 부른다. 동작 자체가 가벼워 1일차 비용이 거의 0이므로 미룰 이유가 없다.
  • 트레이드오프: 1단계가 단일 tenant라 합성·체인이 자명하게 동작하나, 규약을 미리 고정하는 약간의 선행 설계. 무재작업 대가로 수용.

A의 출하속도 골격 위에, C의 책임경계(전역 fail-closed authority·confidence+status 게이팅)와 층화(Capture/LIFT·NORMALIZE·scenarios·ContractBinder)를 "필드·단계·훅"으로 1일차 접목하고, B의 타입 규율(unknown≠null·단일 ir-schema)을 흡수했다. 적대적 검토가 지적한 횡단 실행계약 공백 — status가 게이트까지 못 닿던 fail-closed, 주입 경로 없던 계약 정합성, 규약 없던 nodeId, 모순된 Promoter↔facts 순서, 미뤄둔 dedup·exit-code·prevHash·tenant 합성 — 을 전부 메우되, 모든 보강이 "스키마 필드/파이프라인 단계/결정적 규약" 추가라 인터페이스 동결 원칙을 깨지 않는다. 미래 비용은 스키마/인터페이스에 1일차 투자해 막고, 현재 비용은 인프라를 2단계로 미룬다 — 빈 L3, 순수함수 룰(서버화 예약), 바인딩·계약 슬롯(데이터정합성 예약), 심각도/rollout 분리(거버넌스 예약), 어댑터=Capture(입력확장 예약), tenantId 타입강제+합성함수(멀티테넌트 예약), authority 필수(SaMD 방어), prevHash 체인(감사 무결성 예약). 이 경계들이 1→3단계 무재작업을 물리적으로 보증하고, 진짜 1단계 롱폴 리스크인 Semantic Promoter를 1순위로 격상해 명세했다.
